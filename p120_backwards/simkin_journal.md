# Simkin Glider Gun Building Journal

Let's try to build a simkin's glider gun at a 5 lane offset using an "Active Region Elbow" aka, just sending a stream of gliders in whatever way makes progress.

First, we need a slow salvo recipe so that we have some way to measure progress. We aren't planning to build the gun using zero degree gliders, but the script assumes that we have a recipe, so let's make one with pslmake.

Here's the seed, found by eeee:
```bash
x = 33, y = 29, rule = B3/S23
23bo$21b3o$20bo$20b2o5$24b2o5b2o$24b2o5b2o2$27b2o$27b2o2$8b2o$8b2o$4b
2o$4b2o11bo$16bobo$2o14bobo$2o7b3o5bo6$14b2o$13bobo$15bo!
```

```bash
pslmake/pslmake.py -i single-channel-toolit/snark-maker/p120_backwards/simkin-seed.rle
```

pslmake gives a 43 glider recipe, including the final glider that activates the gun. See simkin-seed-slow-salvo.rle. That final glider is important, because the last glider in the recipe defines our single channel lane as well.

Now, we need to generate the recipe intermediates file. This is a database showing all of the intermediate stages you can reach when following the slow salvo. The program uses these to check how much progress you've made. Again, we probably won't use most of them since we're going to try building the final stage directly instead of incrementing through slow gliders, but we'll have them available.

```
uv run snark.py recipe-intermediates -i p120_backwards/simkin-seed-slow-salvo.rle -o results/simkin-stages.sqlite
```

This saves the database in simkin-stages.sqlite, and it outputs a stamp collection showing the stages. See simkin-seed-recipe-intermediates.rle for the stamp collection.

From now on when we run the `optimize` command, which runs a search, we will pass in `-r results/simkin-stages.sqlite` to copy the intermediates into the search.

Now, we can run our first search!

```bash
uv run snark.py optimize -r results/simkin-stages.sqlite -o results/simkin-search1.sqlite -n 600 --partial-range=42
```

I chose a `--depth` of 600 -- aka, all unique streams that sum to 600! Unless we run for a week, we won't finish. That's ok -- the databases store the queue, and we can Ctrl+C exit at any time, then run the same command again to resume right where we left off if we want to search more.

I set `--partial-range` to 42. This will limit the search to only consider intermediate matches in our intermediate collection that have 42 gliders sent so far, aka, the finished pattern.

The other arguments are default:

- --gen-options=90-255: consider glider spacings of 90-255
- --n-processes=20: run 20 worker processes
- --min-offset-block-lane=0
- --depth-range=-100-100
- --partial-progress-factor=0.1
- --live-view-depth=inf

See snark.py for details on those.

The optimize command will print a status update every 10 seconds so you have an idea how much progress it has made:

```
121.27/s, 264.30 avg/s, 7,699,286 done, 435-600 gens, 1,992/162,774,355 pending, 0x0, -10.11x1, infxinf A (0), inf overlap (0), inf pop (0)
```

KeyL
1. The number of streams processed per second in the last 10 seconds
2. The average streams/s since the current invocation started
3. The total number of streams processed since the current invocation started
4. The current depth of search. 435-600 indicates that we are taking streams with a sum of 345 from the queue, and we are considering follow-up gliders of 90-255.
5. There are 1992 left in the queue for the 435-600 batch, and 162 million overall (and this is growing -- this is why we won't be finishing this search)
6. Our best full match, and how many results have reached that far. Since we limited the search to step 42, we have no full matches.
7. This is what we care about in this search! Our best partial_intermediate_log_prob. This number is based on the rarity of the missing pieces of a partial match and how good that partial match is. We see that there's one result with a log prob of -10. When we first started, the best partial log prob was -17, so we've found a much better match.
8. The area of the ash cloud (but only considers for full matches)
9. The lowest overlapping population (based on separating them by depth) between a full match and the ash cloud.
10. The lowest population of a full match.

We see that the -10 log prob result finds a match for the eater 1!

That's a rare find, so let's take that result and continue from there using setup-next-search. We pass a SQL query or fragment for which results to take as starting points for the next search.

```bash
$ uv run snark.py setup-next-search -i results/simkin-search1.sqlite -o results/simkin-search2.sqlite -q 'select * from r order by partial_intermediate_log_prob desc limit 1'

Transferred 1 results as starting_points.
```

And then we can starting searching there too. Our result had a partial_intermediate_shift of 21, so we can limit the depth range to only search patterns shifted by 21.

```bash
uv run snark.py optimize -r results/simkin-stages.sqlite -o results/simkin-search2.sqlite -n 600 --partial-range=42 --depth-range=21
```

This searches like 10x faster because it's not considering 200 depths for every result. (This is an improvement to make in the script).

This ended up taking only 40 minutes to complete, which is not what I expected for a depth 600 search.  There were only about 6,500,000 streams searched total, while there were 160 million queued and rising in our previous search. That's because the search prunes equivalent branches.
